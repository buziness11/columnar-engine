\section{Введение}
\subsection{Описание предметной области}
Колоночные движки являются популярным решением для исполнения SQL запросов к таблицам,
которые не требуют функционала добавления/удаления строк.
У них есть свой колоночный формат(теперь данные лежат столбцами, а не строками),
что минимизирует количество cache miss’ов и чтений с диска при ответе на запрос.
\par
Для оценки производительности
движков предлагается использовать ClickBench.
Это хороший способ оценить эффективность движков, бенчмарк прогоняет запросы на таблице,
которая содержит порядка 100 столбцов и около 90 миллионов строк,
при этом все данные анонимизированы и основаны на работе популярного сервиса,
что делает условия максимально приближенными к реальным.


\subsection{Постановка задачи}
Нужно Пплучить таблицу hits.csv, создать свой колоночный формат на основе полученной таблицы,
реализовать колоночный движок, работающий с новым форматом, который сможет успешно проходить
бенчмарк ClickBench.


\subsection{Структура работы}
В рамках проекта был создан класс для чтения и записи csv файла а также
класс для чтения и записи колоночного формата.
Связь между ними осуществляется через класс batch
(часть файла, которая помещается в оперативную память).
Этот набор классов позволяет читать csv файл, преобразовывать его
в колоночный формат и наоброт за пару простых шагов.
\par


\section{Обзор литературы}
Так как мы нацелены именно на прохождение бенчмарка ClickBench, то посмотрим на
лидеров по производительности на сайте бенчмарка \cite{clickbench}.
Результаты запуском на сайте представлены на различных машинах, как с cold, так и warm режимами.
Выберем тройку лидеров(все они являются колоночными).
ClickHouse \cite{clickhouse} Parquet \cite{parquet} и DuckDB \cite{duckdb}.
Выделим особенности каждого из движков.
\par
ClickHouse \cite{clickhouse} использует собственные форматы хранения
(семейство движков MergeTree), сжатие и векторные инструкции процессора
для обработки данных.
\par
DuckDB{duckdb} – это встраиваемая (in-process) база данных. У неё нет сервера,
она работает внутри процесса.
Главная особенность – векторизованный движок выполнения запросов,
который работает на одной машине.

\par
Parquet \cite{parquet} – это колоночный формат хранения данных,
который содержит метаданные (статистику min/max),
что позволяет движкам читать только нужные части файла (Predicate Pushdown).
\par
У всех трёх движков есть свои особенности, которые позволяют им быть производительными.

\section{Описание предлагаемого метода}
\subsection{Структура классов}
class Schema - содержит информацию о названиях и типах столбцов таблицы.

\par
class Column - содержит данные одного столбца таблицы,
представляет из себя $std::variant<std::vector<type_1>, ...>$
имеет метод $TranslateTo$ который переводит колонку из одного типа в другой.

\par
class Batch - содержит часть таблицы,
которая помещается в оперативную память,
хранит данные как $std::vector<Column>$ и держит рядом класс $Schema$,
это сделано для того чтобы в батч можно было подставить другую схему,
что вызовет трансформацию столбцов($TranslateTo$) в нужные типы.

\par
class CSV-Reader - читает csv файл и возвращает batch строк согласно RFC-4180  \cite{rfc4180}.

\par
class CSV-Writer - принимает batch дает новую схему в которой типы это строка, записывает в csv файл согласно RFC-4180\cite{rfc4180}

\par
class BZN-Reader - читает колоночный формат и возвращает batch

\par
class BZN-Writer - принимает batch и записывает в колоночный формат (при инициализации создает массив с разметкой батчей, который заполняется по ходу записи)

\par
Для более глубокого понимания всей структуры и методов можно обратится на GitHub \cite{GitHub}
\subsection{Колоночный формат bzn}

На данный момент поддерживается два типа данных: int64\_t, string.
Первый записывается как последовательность 8 байт в little-endian формате.
Второй записывается как последовательность байт(символов), описывающая нашу строку,
после чего идет $\texttt{'\textbackslash x1F'}$ – специальный символ, демонстрирующий конец строки.

\par
Скажем, что количество колонок в нашей таблице это m и поговорим о
роли batch’a в bzn формате:
это набор колонок, соотвествующий какому-то количеству подряд идущих строк из таблицы,
то есть вся исходная таблица представима в виде $batch_1 \sqcup batch_2\sqcup ... \sqcup batch_k$, где $k$ –
количество batch'ей, зависящее от количества строк в таблице и размера
batch'a(на данный момент размер определятеся как количество, хранящихся в batch'e).

\par
batch в bzn формате лежит как: [метаданные][данные].
Метаданные хранятся как подряд идущие int64\_t, которые
описывают позиции в файле, с которых начинается каждая колонка
$poscolumn_1, poscolumn_2,..., poscolumn_m$.
После метаинформации лежат сами данные колонок,
с позиции $poscolumn_i$ в файле лежит $i$-ая колонка(о том как хрянятся различные типы уже говорили).
Далее поговорим о метаданных всего файла. Из которых будет очевидно количество колонок в таблице, а значит и задача чтения батча очевидна.

\par
Формат bzn выглядит следующим образом: \newline
[размер метаданных][$batch_1$][$batch_2$]...[$batch_k$][метаданные]
$batch_i$ хранится как сказано выше. \par
Метаданные в конце файла хранятся как: \newline
[имена столбцов][типы столбцов][разметка батчей]
\par
Где имена столбцов – это последовательность строк,
разделенных символом $\texttt{'\textbackslash x1F'}$,
после которых идет символ $\texttt{'\textbackslash x1E'}$ говорящий об окончании этой секции метаданных.
Типы столбцов хрянятся аналогично их строковыми представлениями.
После них также идет символ $\texttt{'\textbackslash x1E'}$
и начинается разметка батчей,
которая представляет собой последовательность int64\_t,
каждый из которых говорит о позиции в файле,
с которой начинается соответствующий батч.

\par
% метаданные пердставляют собой schema, которая хранится как [набор строк имен]’\x1E’[набор строк типов]’\x1E’[разметка batch’ей]

\section{Заключение}
На данном этапе я могу создать колоночный формат из csv файла и обратно.
Мне достаточно прочитать схему, создать CSVReader, читать из него батчи пока возможно,
перекидывать их в MyFormatWriter и записывать в файл.
